# Topic 8: Software Engineering & OOP Design (50 Q)

<details>
<summary><b>Q: Software Engineering definition?</b></summary>
<blockquote><b>Answer:</b> Systematic approach to software development using engineering principles</blockquote>
</details>

<details>
<summary><b>Q: Waterfall Model phases?</b></summary>
<blockquote><b>Answer:</b> Requirements → Design → Implementation → Testing → Deployment (sequential)</blockquote>
</details>

<details>
<summary><b>Q: Iterative Model?</b></summary>
<blockquote><b>Answer:</b> Repeated cycles: design → implement → test. Incremental delivery</blockquote>
</details>

<details>
<summary><b>Q: Agile Model?</b></summary>
<blockquote><b>Answer:</b> Flexible, iterative. Short sprints, continuous feedback, adapts to change</blockquote>
</details>

<details>
<summary><b>Q: Spiral Model?</b></summary>
<blockquote><b>Answer:</b> Risk-driven. Combines iterative + waterfall. 4 phases: planning, design, build, evaluate</blockquote>
</details>

<details>
<summary><b>Q: Use case?</b></summary>
<blockquote><b>Answer:</b> Describes interaction between user and system. Shows system behavior</blockquote>
</details>

<details>
<summary><b>Q: UML (Unified Modeling Language)?</b></summary>
<blockquote><b>Answer:</b> Standard notation for modeling software. Class, sequence, use case diagrams</blockquote>
</details>

<details>
<summary><b>Q: Class diagram?</b></summary>
<blockquote><b>Answer:</b> Shows classes, attributes, methods, relationships. Static structure</blockquote>
</details>

<details>
<summary><b>Q: Sequence diagram?</b></summary>
<blockquote><b>Answer:</b> Shows object interactions over time. Message passing sequence</blockquote>
</details>

<details>
<summary><b>Q: Design pattern?</b></summary>
<blockquote><b>Answer:</b> Reusable solution to common problem. Examples: Singleton, Factory, Observer</blockquote>
</details>

<details>
<summary><b>Q: Singleton pattern?</b></summary>
<blockquote><b>Answer:</b> Only one instance of class. Global access point</blockquote>
</details>

<details>
<summary><b>Q: Factory pattern?</b></summary>
<blockquote><b>Answer:</b> Creates objects without specifying exact classes</blockquote>
</details>

<details>
<summary><b>Q: MVC (Model-View-Controller)?</b></summary>
<blockquote><b>Answer:</b> Model (data), View (UI), Controller (logic). Separates concerns</blockquote>
</details>

<details>
<summary><b>Q: Analysis vs Design?</b></summary>
<blockquote><b>Answer:</b> Analysis: Understanding requirements (WHAT). Design: Planning solution (HOW)</blockquote>
</details>

<details>
<summary><b>Q: SOLID principles?</b></summary>
<blockquote><b>Answer:</b> Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion</blockquote>
</details>

<details>
<summary><b>Q: Single Responsibility Principle?</b></summary>
<blockquote><b>Answer:</b> One reason to change. Class has one responsibility</blockquote>
</details>

<details>
<summary><b>Q: Open/Closed Principle?</b></summary>
<blockquote><b>Answer:</b> Open for extension, closed for modification</blockquote>
</details>

<details>
<summary><b>Q: Cohesion?</b></summary>
<blockquote><b>Answer:</b> How closely module elements belong together. High cohesion = good</blockquote>
</details>

<details>
<summary><b>Q: Coupling?</b></summary>
<blockquote><b>Answer:</b> Interdependence between modules. Low coupling = good</blockquote>
</details>

<details>
<summary><b>Q: Refactoring?</b></summary>
<blockquote><b>Answer:</b> Improve code structure without changing functionality</blockquote>
</details>

<details>
<summary><b>Q: Code review?</b></summary>
<blockquote><b>Answer:</b> Peers examine code for quality, correctness, standards</blockquote>
</details>

<details>
<summary><b>Q: Unit Testing?</b></summary>
<blockquote><b>Answer:</b> Test individual modules/functions</blockquote>
</details>

<details>
<summary><b>Q: Integration Testing?</b></summary>
<blockquote><b>Answer:</b> Test module interactions</blockquote>
</details>

<details>
<summary><b>Q: System Testing?</b></summary>
<blockquote><b>Answer:</b> Test complete system against requirements</blockquote>
</details>

<details>
<summary><b>Q: Acceptance Testing?</b></summary>
<blockquote><b>Answer:</b> Test with user/client. Validates requirements met</blockquote>
</details>

<details>
<summary><b>Q: Black Box Testing?</b></summary>
<blockquote><b>Answer:</b> Test functionality without knowing internal code</blockquote>
</details>

<details>
<summary><b>Q: White Box Testing?</b></summary>
<blockquote><b>Answer:</b> Test with knowledge of internal code. Coverage analysis</blockquote>
</details>

<details>
<summary><b>Q: Test-Driven Development (TDD)?</b></summary>
<blockquote><b>Answer:</b> Write test → Fail → Write code → Pass → Refactor</blockquote>
</details>

<details>
<summary><b>Q: CI/CD?</b></summary>
<blockquote><b>Answer:</b> CI: auto build+test on commit. CD: auto deploy to production</blockquote>
</details>

<details>
<summary><b>Q: Builder pattern?</b></summary>
<blockquote><b>Answer:</b> Construct complex objects step-by-step. Separates construction from representation</blockquote>
</details>

<details>
<summary><b>Q: Dependency Injection?</b></summary>
<blockquote><b>Answer:</b> Provide dependencies externally. Improves testability, flexibility</blockquote>
</details>

<details>
<summary><b>Q: REST API?</b></summary>
<blockquote><b>Answer:</b> Uses HTTP methods (GET, POST, PUT, DELETE). Stateless</blockquote>
</details>

<details>
<summary><b>Q: Microservices?</b></summary>
<blockquote><b>Answer:</b> Small, independent services. Communicate via APIs. Scalable, maintainable</blockquote>
</details>

<details>
<summary><b>Q: Monolithic architecture?</b></summary>
<blockquote><b>Answer:</b> Single large application. Tightly coupled. Difficult to scale</blockquote>
</details>

---
